## 2026-07-07 — #165 sub-item 3: budget period navigation (1M/3M/6M/12M)

### Resolved
- Segmented control (1M/3M/6M/12M) + ‹ › arrows above the Budget card (`BudgetPeriodCard`, new, in `Home.tsx`). 1M keeps `BudgetDashboard` exactly as-is (reused unchanged, just fed a navigable month/year instead of always "now"); 3M/6M/12M swap it for `<Chart chart_type="line">`, reusing Chart.tsx's existing generic contract per the issue's explicit instruction — no new chart component. One endpoint, `GET /api/home/budget-period?period=&month=&year=`, not one per window; `period=month` reuses `get_budget_status()` unchanged (architecture.md rule 20 — no new calculation), the trend modes reuse a new client method (see bug below).
- **Bug found live #1 (backend, real, not cosmetic):** the first trend implementation called `client.get_monthly_stats()` in a loop, one call per month — each call does its own full Actual Budget login + `download_budget()`. A 12M request means 12 logins in a few hundred ms, which reliably tripped Actual Budget's own login rate limit (`429 Too Many Requests`), which then failed *every* request (including plain 1M) for a cooldown window. Fixed with a new `client.get_monthly_totals_batch(months: list[tuple[int,int]])` — one `download_budget()` session, loops internally reusing `_compute_monthly_totals()`. The pre-existing `get_spending_trend` chat tool has the exact same loop-of-logins pattern and hasn't hit this yet only because nothing exercises a 12-month window through it today — noted as a latent bug, not fixed here (out of scope for #165), should move onto the batch method next time that tool is touched.
- **Bug found live #2 (frontend, real, not cosmetic):** switching period (e.g. 3M → 12M) updated the nav label correctly but left the chart showing the *previous* period's stale data. Root cause: `Chart`'s internal `useChartRefetch` hook only uses its `title`/`data` props as a `useState` *initial* value — it needs a changed `key` to remount and pick up new data. The first fix (`key={`${period}-${month}-${year}`}`) looked right but wasn't: `period`/`month`/`year` update on an earlier render (the moment the button is clicked) than the trend data itself (which only lands after the fetch resolves) — so the remount happened one render too early, and the later props-only update was silently ignored by Chart's internal state. Fixed by keying on a `requestId` stored *inside* the same state object as the trend data, so the key and the data it identifies always change together in the same `setTrend` call. Also added a stale-response guard (`requestIdRef`, ignore a response if a newer request has since started) as defense-in-depth against Actual Budget's single request lock queuing requests out of the order they were fired.
- Verified live via Playwright across the full matrix: 1M → 3M → previous-period (3M) → 12M → 1M → previous-period (1M), checking both the nav label and the actual rendered chart/table data at each step, not just the label (the bug above would have passed a label-only check).

### Files modified
- backend/core/actual_client/client.py (`get_monthly_totals_batch`, new), backend/api/home.py (`get_budget_period`, new endpoint)
- frontend/src/lib/api.ts (`getBudgetPeriod`, `BudgetPeriod`/`BudgetPeriodMonth`/`BudgetPeriodTrend`), frontend/src/components/Chart.tsx (exported `LineData`), frontend/src/pages/Home.tsx (`BudgetPeriodCard`, new)

### Lessons
- A `key` fix that "looks right" (ties to the values that conceptually changed) can still be wrong if those values update on a *different render* than the data being keyed — always verify by checking the rendered data itself after a rapid sequence of changes, not just that *some* UI element (like a label) updated. This is worth a general callout, not just this feature.
- Looping single-purpose client methods (each opening its own session) to build a multi-period aggregate is a trap that's easy to fall into when the single-period method already exists and "just works" for the 1-item case — check whether N sequential calls means N logins before reaching for a loop.

### Lessons
- Fixing a stale/wrong tool description (schema `description` field, not just the missing system-prompt bullet) matters just as much — an LLM reading "no confirmation needed" on a tool that actually returns a confirmation card is exactly the kind of prompt/reality mismatch the #160/#166 gotcha already warns about, just in a different field.
- A z-index collision between a modal and persistent chrome (bottom nav) can sit unnoticed for a long time if nothing interactive ever lands in the affected region — it only became visible the moment a real tap target (not just text) was placed near the bottom of a popup.
