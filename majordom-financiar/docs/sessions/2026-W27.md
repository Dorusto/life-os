## 2026-07-02 — Issue #103: CSV import currency conversion (crypto.com)

### Resolved
- **Issue #103** — turned out not to need real FX conversion: crypto.com's export already provides a "Native Amount"/"Native Currency" column pair, already converted to the account's home currency (EUR). The saved profile (`col_amount: "Amount"`) was pointing at the raw foreign-currency column instead. Real case confirmed: Porkbun.com charged 9.78 EUR, but "Amount" (USD) showed 11.08 — "Native Amount" correctly shows -9.78.
- `CsvProfileDetector`'s LLM prompt (`_DETECT_PROMPT` in `detector.py`) now explicitly instructs: when both a foreign-currency amount column and an already-converted native/home-currency amount column exist, always prefer the converted one for `col_amount` and leave `col_currency` empty.
- Corrected the already-saved crypto.com profile in `memory.db` directly (`col_amount` → `"Native Amount"`) so the fix applies immediately without waiting for LLM re-detection.

### Verified live
- Reimported the real crypto.com export (`tests/CSV/card_transactions_record_20260702_182738.csv`) — Porkbun row now previews at 9.78 EUR (was 11.08)
- Reimported a Revolut export with a RON-denominated transaction settled directly in EUR (`tests/CSV/account-statement_2026-06-01_2026-07-02_en-gb_7c0627.csv`) — unaffected, still correct (Revolut has no native/foreign column pair, single `Amount`+`Currency`, both already EUR)

### Files modified

### Lessons

## 2026-07-02 — Issue #118: CSV import account auto-suggestion

### Resolved
- Found while re-testing #103 against the real crypto.com export: the account selector always showed "No account matched" for "crypto.com" vs. account "Crypto card" — existing matching logic (`ImportPage.tsx` + `CsvImportCard.tsx`, duplicated identically in both) was plain substring matching (`a.includes(b) || b.includes(a)`), which fails whenever neither string fully contains the other.
- Replaced with word-token overlap matching (`matchAccountBySource()`, new shared `frontend/src/lib/csvImportUtils.ts`): splits both strings on non-alphanumeric characters, keeps tokens ≥3 chars, matches if any token overlaps. Catches "crypto.com" → "Crypto card" via the shared "crypto" token.
- Extracted into a shared helper since the exact same (buggy) logic was duplicated in both import UI components.

### Verified
- Unit-checked the matching function directly with the real account list (ING current, BUNQ Current, Crypto card, etc.): crypto.com → Crypto card ✅, ING → ING current ✅, BUNQ → BUNQ Current ✅
- `tsc --noEmit` clean, `majordom-web` rebuilt

### Files modified
- `frontend/src/lib/csvImportUtils.ts` — new: `matchAccountBySource()`
- `frontend/src/pages/ImportPage.tsx` — use shared helper
- `frontend/src/components/CsvImportCard.tsx` — use shared helper

### Lessons
- Token overlap can false-positive on generic words (e.g. a hypothetical "Unknown Bank XYZ" source matching an account literally named "Bank Name" via the shared "bank" token) — acceptable since this only preselects a dropdown value the user can still change before confirming, never auto-executes.

### Unresolved

## 2026-07-02 — Near-duplicate detection for CSV import (date+merchant match, different amount)

### Resolved
- Root cause of the Porkbun/Revolut-deposit reports above: exact-hash dedup (`SHA256(date+merchant+amount)`) can never catch a transaction re-imported with a *corrected* amount — different amount means different hash, so it silently looks brand new and would create a duplicate on confirm.
- Added a secondary check in `preview_csv`: `_fetch_preview_data()` now also builds a `(date, payee name lower) -> [existing amounts]` index across all AB transactions. For any CSV row that isn't an exact duplicate, if a transaction already exists in AB on the same date with the same merchant but no matching amount, the row is flagged `possible_duplicate=True` with `existing_amount` set — shown in both import UIs as a yellow warning icon + tooltip (ImportPage.tsx also shows an inline text line). Purely advisory — does not block import, still fully user-editable per the confirmation-card rule.

### Verified live
- Created a transaction manually (simulating a historical wrong-amount import: "Porkbun Test 2", 2026-06-25, €11.08), then previewed a synthetic crypto.com-format CSV with the same date+merchant at the corrected €9.78 (Native Amount). Response: `duplicate: false, possible_duplicate: true, existing_amount: 11.08` — exact reproduction of the real Porkbun case reported live.

### Files modified
- `backend/api/csv_import.py` — `_fetch_preview_data()` returns near-dup index; `ImportRowPreview` gains `possible_duplicate`/`existing_amount`; `preview_csv` computes the flag per row
- `frontend/src/lib/api.ts` — `ImportRow` type extended
- `frontend/src/pages/ImportPage.tsx` — warning icon + inline text for possible-duplicate rows
- `frontend/src/components/CsvImportCard.tsx` — warning icon (tooltip only, compact chat card layout)

### Lessons
- Two historical CSV rows reported live (Porkbun @ old wrong amount, 2 Revolut transfer deposits) were not caught by exact dedup because they were created in AB *before* the #101/#102/#103 fixes landed (old/missing `financial_id`) — a one-time data-migration side effect of fixing historical wrong data, not a new bug. This near-duplicate check is what makes that class of situation visible instead of silent, going forward.

### Bug found live in production, same day
- The initial version only flagged `possible_duplicate` when the existing amount *differed* from the CSV's. Tested live against the real Porkbun row in production: not flagged. Root cause — `financial_id` is fixed at creation time; the real Porkbun transaction had already been hand-corrected in AB (amount edited 11.08 → 9.78 directly in the UI), so the *displayed* amount now matched the CSV, but `financial_id` still held the old hash. Fix: flag on any (date, merchant) match regardless of amount equality. Reverified live against both sub-cases (amount matches / amount differs) plus a regression check (20-row exact-duplicate CSV — 0 false positives). Commit `f486786`.

## 2026-07-03 — Categorization bug batch (#104, #106, #107, #108)

### Resolved
- **#104 (unify categorization tools):** `propose_categorize_by_payee` and `propose_categorize_with_rule` were near-identical, only differing in optional rule creation — the LLM could silently pick the no-rule variant on an ambiguous request. `propose_categorize_with_rule` already supported an optional `create_rule` override, so it's now the *only* categorization tool. Removed `propose_categorize_by_payee` entirely (registry schema + dispatcher, `actual_budget.py` function, `_PROPOSAL_TOOLS` entry, the dead `categorize_by_payee` branch in `category_actions.py`). Added an editable payee input to the remaining card (`CategoryActionCard.tsx`), matching what the removed tool's card offered. Verified live: only one categorize tool registered; `propose_categorize_with_rule` returns the full card (payee, count, category, rule).
- **#107 (inconsistent card vs. direct execution):** reproduced the exact original phrase ("transferurile catre xtb sunt investitii") live, 3 times, 2 phrasings — always returned a proper confirmation card, never direct execution. Root cause was almost certainly the same tool-selection ambiguity as #104; closed as fixed by #104's unification.
- **#108 (setup balance card forces adjustment):** turned out the underlying mechanism already supported deferring — `adjust_account_balance()` no-ops (creates nothing) when the submitted value matches the current balance, so leaving a field untouched was already safe. The real gap was UX clarity: a pre-filled number looks verified even when it isn't, no visible way to say "I don't know, skip this". Added an explicit **Skip** toggle per account in `SetupBalancesCard.tsx` — disables the input and forces the current balance to be sent regardless of what's typed, removing any ambiguity about whether skipping is safe.

### Investigated, left open
- **#106 (card disappears on navigation):** chat `messages` state is lifted to `App.tsx`/`Layout`, above `<Routes>` — in-app `NavLink` navigation (Home → Chat) shouldn't unmount it, and an existing fix (`bb37852`, 2026-06-01) already guards active cards against being overwritten by `loadChatHistory()` on window-focus reload. Architecturally, simple in-app navigation should already be safe. Real gap found: proposal/card-type messages are explicitly excluded from `saveChatHistory()` (only the last user+assistant text exchange is persisted server-side) — if the *page itself* gets reloaded (mobile PWA killed and restarted while backgrounded, not just switched-to within the SPA), the card is genuinely unrecoverable. User wasn't sure which scenario triggered the original report; left open. If it recurs, the PWA-restart case needs server-side persistence of structured card JSON, not just role+content strings — bigger schema change.

### Verification method
No pytest suite for these flows — verified live against the local docker-compose stack (`majordom-api`/`majordom-web`/`actual-budget` on this machine, test data only) by calling `execute_tool()` directly in the running container and hitting `/api/chat` with curl, streaming the raw JSON response.

### Lessons
- Two of three original bug reports (#107, #108) turned out to already be non-issues or side-fixed once actually re-tested against current code — the vault findings were accurate for the day they were written (2026-07-01/02), but code moves fast enough that re-verifying live before implementing is essential, not optional. Don't trust a month/day-old finding without reproducing it first.
- The license-plate regex in `scripts/check-private-data.sh` false-positives on ordinary Tailwind CSS classes when a short letter-only class sits right next to a numeric-width class next to another letter-only class (letter+digits+letters pattern). Fixed by reordering classes so the pattern breaks, not by bypassing the hook — worth knowing if it happens again, don't reach for `--no-verify`.
