## 2026-07-02 — Near-duplicate detection for CSV import (date+merchant match, different amount)

### Resolved
- Root cause of the Porkbun/Revolut-deposit reports above: exact-hash dedup (`SHA256(date+merchant+amount)`) can never catch a transaction re-imported with a *corrected* amount — different amount means different hash, so it silently looks brand new and would create a duplicate on confirm.
- Added a secondary check in `preview_csv`: `_fetch_preview_data()` now also builds a `(date, payee name lower) -> [existing amounts]` index across all AB transactions. For any CSV row that isn't an exact duplicate, if a transaction already exists in AB on the same date with the same merchant but no matching amount, the row is flagged `possible_duplicate=True` with `existing_amount` set — shown in both import UIs as a yellow warning icon + tooltip (ImportPage.tsx also shows an inline text line). Purely advisory — does not block import, still fully user-editable per the confirmation-card rule.

### Verified live
- Created a transaction manually (simulating a historical wrong-amount import: "Porkbun Test 2", 2026-06-25, €11.08), then previewed a synthetic crypto.com-format CSV with the same date+merchant at the corrected €9.78 (Native Amount). Response: `duplicate: false, possible_duplicate: true, existing_amount: 11.08` — exact reproduction of the real Porkbun case reported live.

### Files modified
- `backend/api/csv_import.py` — `_fetch_preview_data()` returns near-dup index; `ImportRowPreview` gains `possible_duplicate`/`existing_amount`; `preview_csv` computes the flag per row
- `frontend/src/lib/api.ts` — `ImportRow` type extended
- `frontend/src/pages/ImportPage.tsx` — warning icon + inline text for possible-duplicate rows
- `frontend/src/components/CsvImportCard.tsx` — warning icon (tooltip only, compact chat card layout)

### Lessons
- Two historical CSV rows reported live (Porkbun @ old wrong amount, 2 Revolut transfer deposits) were not caught by exact dedup because they were created in AB *before* the #101/#102/#103 fixes landed (old/missing `financial_id`) — a one-time data-migration side effect of fixing historical wrong data, not a new bug. This near-duplicate check is what makes that class of situation visible instead of silent, going forward.

### Bug found live in production, same day
- The initial version only flagged `possible_duplicate` when the existing amount *differed* from the CSV's. Tested live against the real Porkbun row in production: not flagged. Root cause — `financial_id` is fixed at creation time; the real Porkbun transaction had already been hand-corrected in AB (amount edited 11.08 → 9.78 directly in the UI), so the *displayed* amount now matched the CSV, but `financial_id` still held the old hash. Fix: flag on any (date, merchant) match regardless of amount equality. Reverified live against both sub-cases (amount matches / amount differs) plus a regression check (20-row exact-duplicate CSV — 0 false positives). Commit `f486786`.

## 2026-07-03 — #87 (copy last month's budget) + #125 fix + rollover toggle (#124 partial)

### Resolved
- **#87:** `propose_budget_copy(month="")` fetches the previous month's per-category budgeted amounts via a new `get_budget_copy_source()` client method, returns an editable card (grouped by category group, pre-filled amounts) — nothing written until confirmed. Confirm applies each (possibly user-edited) amount via the existing `set_budget_amount()`.
- **#125 (fixed as part of #87):** goal-template categories are excluded from the copy — detected via `"#template"` in the category's Notes entry (the `Notes` table, keyed by category id — categories have no `notes` column directly).
- **Proactive variant, added on request:** `_check_budget_copy_nudge()` in the daily digest — first 5 days of the month, nudges if last month had a real budget but the current month looks empty. New notification rule `budget_copy_nudge` seeded in `main.py`.
- **Rollover overspending, added on request (#124 partial):** `propose_set_budget_carryover(category_name, enabled, month="")` toggles the `carryover` column on ZeroBudgets/ReflectBudgets — the same field AB's own UI writes. Confirmed via `actualpy` source that `create_budget()`'s `carryover` kwarg only applies when creating a brand-new budget row (silently ignored if a budget already exists for that month, the common case) — `set_budget_carryover()` sets the field directly on the existing row instead.

### Two real bugs found and fixed while building this
1. **`Categories.goal_def`/`template_settings` are not reliable "has a goal template" signals.** They looked plausible (a newer structured "Budget Automations" system was mentioned in earlier research) but verified live: `template_settings` returns `{'source': 'notes'}` on literally every category regardless of whether it has a template, and `goal_def` stayed `None` even on a category with a real `#template` note. Removed both checks, kept only the Notes-text check that's actually confirmed to work.
2. **The `zero_budgets`/`reflect_budgets` SQL lookup filtered `WHERE id LIKE '{yyyymm}%'`, assuming every budget row's `id` follows a `"{month}-{category_id}"` convention.** It doesn't — `actualpy`'s `create_budget()` generates a random UUID `id` for brand-new rows. Any budget set through Majordom's own tools (not the real AB app) was silently invisible to this filter. This pattern was copied across **three call sites** — `get_budget_status()`, `get_home_data()` (Home screen), and the new `get_budget_copy_source()` — so it wasn't just breaking the new feature, it's very likely been under-reporting budgeted amounts on the Home dashboard for any category ever budgeted via chat, since M4.12. Fixed all three to filter by the `month` column instead. Found via a direct discrepancy: `set_budget_amount()` reported success (`old_amount: 150.0, new_amount: 150.0`) for Transport, but the copy-source query still showed `0.0` for the same category/month — cross-checked the raw table and found the real row had a random-UUID `id`, invisible to the old filter.

## 2026-07-03 — Home dashboard: rollover categories were disappearing (reported on real production data)

### Resolved
- User reported live, on their real (non-demo) budget: a "Holidays" category funded once with rollover enabled, then relied on for spending across the following month with no fresh monthly allocation — vanished from Home entirely, even though real money was still available.
- Root cause: `get_home_data()` only ever read the current month's *fresh* budgeted amount from `zero_budgets`/`reflect_budgets` (0, since nothing new was allocated) — never the *accumulated* rollover balance.
- Fixed using `actualpy`'s `get_accumulated_budgeted_balance()` — the same value AB's own UI shows as a category's Balance. When a category's fresh monthly amount is 0, fall back to the accumulated balance as the effective "budgeted" amount for display. Naturally solves both halves of the request: shows while a carried-over balance exists, disappears once it's fully spent (no separate "hide it" logic needed).

### Two more bugs caught while implementing
1. The accumulated-balance fallback was initially placed *after* the existing `budgeted==0 and spent==0 -> skip` check — a rollover category with a balance but zero spending yet that month got filtered out before ever reaching the fallback. Moved the fallback earlier.
2. `get_category()` (actualpy) resolves by **name**, not by ID — passing the raw `category_id` string returned `None` internally and silently broke the balance lookup. Had to pass the actual `Categories` object instead (already available from the category fetch earlier in the same function).

### Verified live
- Set "Divertisment" budgeted €500 in June with rollover enabled, explicitly zeroed July's fresh amount. Home correctly showed `budgeted: 544.10` for July (June's ending accumulated balance, including some prior test data).
- Spent the full €544.10 in July → re-checked: accumulated balance correctly dropped to €0.00.

### Files modified
- `backend/core/actual_client/client.py` — `get_home_data()` accumulated-balance fallback

### Lessons
- This bug had been live on the user's real budget, not just the test instance — a good reminder that "works in the demo data" isn't the same as "works for how the user actually uses rollover categories" (funded once, spent gradually, no monthly re-allocation is a normal and common pattern for a category like a trip fund).
- Two `actualpy` helper functions found today (`get_accumulated_budgeted_balance`, and earlier `get_category()`'s name-only resolution) both had non-obvious behavior that only surfaced by calling them directly and checking real output — worth continuing to verify actualpy behavior empirically rather than assuming from function names alone.
